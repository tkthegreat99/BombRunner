붐 러너 (Boom Runner) 기획문서
Clean Rewrite

문서 목적
- 이 문서는 붐 러너의 현재 기획 기준을 처음부터 다시 정리한 단일 기준 문서다.
- 이전 문서의 뒤쪽에 추가되며 생긴 충돌 표현을 제거한다.
- 이 문서에서는 고정 폭탄 시간, 전원 몰살, 라스트 스탠드, ms 단위 생존 판정 같은 이전 방향을 사용하지 않는다.


============================================================
1. 게임 개요
============================================================

붐 러너는 탑다운 3D 멀티플레이 캐주얼 생존 파티 게임이다.

플레이어들은 살아있는 폭탄에게 쫓기며, 폭탄의 타겟이 된 플레이어는 다른 플레이어에게 타겟을 넘기거나, 맵 기믹과 도발, 대시, 아이템을 활용해 살아남아야 한다.

핵심 재미는 다음과 같다.
- 폭탄이 누구를 쫓는지 모두가 명확히 안다.
- 타겟은 폭탄을 피하면서 다른 플레이어에게 태그를 시도한다.
- 비타겟 플레이어는 타겟을 피하거나, 도발로 방해하거나, 아이템/맵 기믹을 활용한다.
- 폭탄이 터질 때는 폭발 범위 안의 모든 사람이 죽는 것이 아니라, 폭탄으로부터 가장 가까운 1명만 쓰러진다.
- 쓰러진 플레이어도 맵 위에서 엉금엉금 기어다니며 코믹한 장애물이 된다.
- 매치는 최후의 1인이 남을 때까지 계속된다.

게임 톤:
- 귀엽고 말랑한 파티게임.
- 폭탄은 무섭지만 코믹해야 한다.
- 실패와 탈락도 억울하기보다 웃겨야 한다.
- 판정은 명확하고 납득 가능해야 한다.


============================================================
2. 기본 경기 구조
============================================================

플레이 인원:
- 기본 2~8명.
- 권장 기본 플레이는 4~6명.
- 커스텀 방에서는 인원과 맵/룰 옵션을 조정할 수 있다.

매치 구조:
- 붐 러너는 매치제를 사용하지 않는다.
- 한 매치는 모든 플레이어가 살아있는 상태로 시작해 최후의 1인이 남을 때까지 이어진다.
- 중간에 다시 살아나는 구조는 없다.
- 별을 획득하는 구조도 없다.
- 쓰러진 플레이어는 다시 생존자로 복귀하지 않는다.
- 최후까지 생존한 1명이 곧 매치 승자다.

기본 진행:
1. 플레이어들이 스폰된다.
2. 첫 폭탄이 생성된다.
3. 폭탄이 생존 플레이어 중 한 명을 타겟으로 설정한다.
4. 폭탄이 타겟을 추격한다.
5. 폭탄이 터질 때마다 폭탄으로부터 가장 가까운 생존 플레이어 1명이 쓰러진다.
6. 쓰러진 플레이어는 생존자 수에서 제외되고, 엉금엉금 기어다니는 장애물 상태로 남는다.
7. 탈락 처리 후 새 폭탄이 생성되고 새 타겟을 설정한다.
8. 생존자가 1명만 남을 때까지 3~7번을 반복한다.
9. 마지막 생존자가 매치 승자가 된다.

매치 승리:
- 최후의 1인이 매치 승자다.
- 별, 매치 점수, 누적 매치 승리 보상은 사용하지 않는다.
- 매치가 끝나면 결과 화면에서 승자와 탈락 순서를 보여준다.


============================================================
3. 플레이어 상태
============================================================

생존 상태:
- 정상 조작 가능.
- 이동, 대시, 태그, 도발, 아이템 사용 가능.
- 폭탄 타겟 후보가 될 수 있다.
- 매치 승리 후보에 포함된다.

타겟 상태:
- 폭탄이 추격 중인 플레이어.
- 머리 위 화살표, 발밑 레이더, 폭탄과의 연결 레이저/라인 등으로 명확히 표시한다.
- 다른 생존 플레이어에게 가까이 접근해 태그하면 타겟을 넘길 수 있다.
- 태그 면역 상태인 플레이어에게는 타겟을 넘길 수 없다.

태그 면역 상태:
- 타겟을 넘겨받은 직후 일정 시간 동안 다시 타겟을 받을 수 없는 상태다.
- 태그 면역은 폭탄/폭발/기믹을 막는 방어막이 아니다.
- 노란 전기 보호막, 버블 실드, 방어막처럼 보이는 연출은 사용하지 않는다.
- 머리 위 원형 게이지로만 표시한다.
- 기본 태그 면역 시간은 3초다.
- 다른 플레이어가 태그 면역 상태의 플레이어에게 태그를 시도하면, 면역 상태 플레이어 머리 위에 “태그 면역 상태입니다!” 문구를 짧게 표시한다.
- 문구가 과도하게 반복되지 않도록 짧은 표시 쿨타임을 둔다.

다운/탈락 상태:
- 폭발 판정으로 선택된 플레이어는 쓰러진 상태가 된다.
- 다운 플레이어는 생존자 수에서 제외된다.
- 다운 플레이어는 폭탄 타겟 후보가 아니다.
- 다운 플레이어는 매치 승리 후보가 아니다.
- 다운 상태에서도 맵 위에 남아 엉금엉금 느리게 이동할 수 있다.
- 다운 상태의 카메라는 기본적으로 본인 캐릭터를 추적한다.
- 관전 전환은 추후 옵션으로 고려할 수 있다.


============================================================
4. 이동 / 충돌 / 대시 / 태그
============================================================

이동:
- 탑다운 3D XZ 평면 이동.
- 점프/Y축 판정 중심의 게임플레이는 사용하지 않는다.
- 기본 이동은 즉각적이고 읽기 쉬워야 한다.

플레이어 충돌:
- 캐릭터들끼리 서로 완전히 겹치면 안 된다.
- 생존 플레이어끼리는 충돌 또는 분리 판정이 있어야 한다.
- 서로의 위치가 겹쳐서 한 덩어리처럼 보이는 상황은 피한다.
- 태그가 가능한 거리까지 접근할 수는 있어야 하지만, 캐릭터 중심이 완전히 겹치면 안 된다.
- 충돌은 과도한 밀치기 싸움이 아니라, 자연스럽게 서로의 몸이 구분되는 정도를 목표로 한다.
- 좁은 통로에서 영구적으로 길막이 되지 않도록 충돌 강도와 통로 폭을 함께 조정한다.
- 다운 플레이어도 밟힘/장애물 판정이 있으므로, 생존자와 다운 플레이어의 상호작용 판정이 필요하다.

충돌 구현 방향:
- 초기 로컬 구현에서는 CharacterController 기반 충돌을 우선 고려한다.
- 플레이어끼리 완전 관통하지 않도록 별도 Push/Separation 로직을 둘 수 있다.
- 물리 Rigidbody로 과하게 튕기는 구조는 피한다.
- 네트워크 단계에서는 Host/Master 기준 위치 보정과 충돌 판정 일관성이 필요하다.

대시:
- 짧은 순간 회피/진입 기술.
- 쿨타임이 있다.
- 도발 중에는 사용할 수 없다.
- 도발 영역 안에 들어가면 사용할 수 없다.
- 다운 상태에서는 사용할 수 없다.

태그:
- 타겟 플레이어가 다른 생존 플레이어에게 가까이 접근하면 타겟을 넘길 수 있다.
- 태그 범위는 몸을 비비는 감각이 나도록 작게 유지한다.
- 단, 태그를 위해 캐릭터가 완전히 겹칠 필요는 없어야 한다.
- 태그 범위와 플레이어 충돌 반경은 함께 밸런싱한다.
- 태그 범위는 밸런스 데이터에서 조정 가능해야 한다.
- 태그 면역 상태인 플레이어에게는 타겟을 넘길 수 없다.

============================================================
5. 도발 시스템
============================================================

도발의 정체성:
- 도발은 버튼 홀드 입력이다.
- 도발은 다른 플레이어를 방해하기 위해 위험을 감수하는 장판형 행동이다.
- 도발은 폭탄 타겟을 가져오는 보상 스킬이 아니다.
- 폭탄이 도발자에게 반응하거나 타겟이 도발자로 바뀌는 것은 리스크다.

도발 입력:
- 버튼을 누르고 있는 동안 도발이 유지된다.
- 버튼을 떼면 도발이 취소된다.
- 취소 후 짧은 후딜이 있다.

도발 중 도발자 상태:
- 이동 불가.
- 대시 불가.
- 아이템 줍기/던지기 불가.
- 도발 영역 생성.
- 폭탄, 폭발, 태그, 기믹 위험에는 그대로 노출된다.

도발 영역 효과:
- 도발 영역 안에 들어온 모든 플레이어는 대시를 사용할 수 없다.
- 현재 타겟뿐 아니라 비타겟 생존자도 도발 영역 안에서는 대시 봉인 상태가 된다.
- 도발 영역에 들어온 플레이어의 Dash UI에는 자물쇠 또는 비활성 표시를 보여준다.
- 도발 영역을 벗어나면 대시 봉인은 해제된다.

폭탄 반응:
- 도발을 오래 유지하거나 폭탄 근처에서 도발하면 폭탄이 도발자에게 반응할 수 있다.
- 이 반응은 타겟 변경 또는 폭탄 분노/가속 같은 위험으로 이어질 수 있다.
- 이는 도발자의 보상이 아니라 도발의 부작용이다.

구현 방향:
- PlayerTauntController 또는 TauntController로 분리한다.
- 도발 영역은 Trigger Collider 또는 거리 계산으로 처리한다.
- 도발 영역 내 Dash 봉인은 PlayerDashController가 상태 플래그를 확인하도록 한다.
- Host/Master가 도발 상태와 영역 효과, 폭탄 반응을 확정한다.


============================================================
6. 폭탄 시스템
============================================================

폭탄:
- 살아있는 폭탄 캐릭터.
- 생존 플레이어 중 한 명을 타겟으로 설정하고 추격한다.
- NavMeshAgent 기반 추격을 사용한다.
- Rigidbody 물리로 밀고 다니는 구조가 아니라, 추격 AI가 중심이다.

폭탄 생존 시간:
- 폭탄이 터지는 시간은 고정 상수가 아니다.
- 폭탄은 평온기, 경고기, 초폭주기 3단계를 가진다.
- 각 단계의 지속 시간은 랜덤 범위에서 결정된다.
- 따라서 폭탄의 총 생존 시간도 매번 달라진다.
- 숫자 타이머 UI는 보여주지 않는다.
- 플레이어는 폭탄의 색상, 표정, 소리, 속도, 이펙트를 통해 위험 단계를 읽는다.

폭탄 단계:
1. 평온기
   - 비교적 느림.
   - 덜 위험해 보임.
   - 귀엽고 낮은 긴장감.
2. 경고기
   - 평온기보다 빠름.
   - 색상, 비프음, 표정이 위험해짐.
3. 초폭주기
   - 가장 빠름.
   - 심장박동, 스파크, 화면 긴장 연출이 강해짐.

단계별 속도:
- 폭탄 기본속도는 평온기 < 경고기 < 초폭주기 순서로 증가한다.
- 분노, 기믹, 다운 플레이어 밟기 가속 등은 단계별 기본속도 위에 별도로 적용될 수 있다.
- 다운 플레이어를 밟아 얻는 가속은 임시 효과이며 영구 속도 증가가 아니다.

폭탄 재생성:
- 폭탄이 터지고 탈락자가 확정되면, 짧은 정리 연출 후 새 폭탄이 생성된다.
- 카메라는 새 폭탄 생성 위치를 짧게 보여준다.
- 새 폭탄은 생존자 중 한 명을 새 타겟으로 설정한다.
- 새 타겟이 확정되면 타겟 표시, 레이더, 연결 연출, 위험 사운드가 갱신된다.
- 이후 매치가 계속 진행된다.


============================================================
7. 폭발 판정
============================================================

핵심 규칙:
- 전원 탈락은 없다.
- 폭발 범위 안에 여러 명이 있어도 탈락자는 1명뿐이다.
- 탈락자는 폭탄으로부터 가장 가까이 있는 생존 플레이어 1명이다.
- 이 규칙을 플레이어가 직관적으로 이해할 수 있도록, 연출상 폭탄 중심에서 링이 퍼져나가 가장 가까운 플레이어에게 먼저 닿는 것처럼 보여준다.
- 링은 판정을 설명하는 시각화다.
- ms 단위 접촉 시간 비교는 사용하지 않는다.
- “라스트 스탠드”라는 용어는 사용하지 않는다.

폭발 후보:
- 폭탄 폭발 범위 안에 있는 생존 플레이어들.
- 다운 플레이어는 탈락 후보가 아니다.
- 폭발 후보가 0명이면 탈락자 없이 폭발 연출만 재생하고 새 폭탄을 생성한다.
- 폭발 후보가 1명이면 해당 플레이어가 탈락한다.
- 폭발 후보가 2명 이상이면 특수 폭발 판정 연출이 나온다.

2명 이상 후보 시 연출:
1. 폭탄 타이머가 끝난다.
2. 슬로우모션 또는 짧은 시간 강조가 들어간다.
3. 폭탄 중심에서 원형 링이 퍼져나간다.
4. 가장 가까운 플레이어에게 링이 먼저 닿는다.
5. 링은 그 지점에서 멈추거나 잠시 정지한다.
6. 해당 플레이어를 셰이더 변화, 외곽선, 하이라이트 등으로 표시한다.
7. 카메라가 줌 아웃되어 폭발 범위와 후보들을 보여준다.
8. 폭발 이펙트가 재생된다.
9. 폭발 범위 안의 플레이어들은 모두 날아간다.
10. 표시된 플레이어 1명만 쓰러진 채로 남는다.
11. 나머지 플레이어들은 날아간 뒤 다시 일어날 수 있다.
12. 새 폭탄이 생성되고 게임이 계속된다.

거리 판정:
- 폭탄 중심과 플레이어 기준점 사이의 XZ 평면 거리를 기준으로 한다.
- Y축 차이는 기본 판정에서 무시하거나 최소화한다.
- 플레이어 기준점은 모든 플레이어에게 동일하게 적용한다.
- CharacterController 중심점 또는 바닥 중심점을 사용할 수 있다.
- 플레이어 반경을 포함할지 여부는 구현/밸런스 단계에서 결정하되, 한 번 정하면 일관되게 적용한다.

동점 처리:
- 같은 거리로 보이는 상황이 생길 수 있다.
- Host/Master가 더 정밀한 XZ 거리 값을 비교한다.
- 그래도 동일하면 결정론적 타이브레이커를 사용한다.
- 숨겨진 랜덤 탈락자는 사용하지 않는다.


============================================================
8. 다운 플레이어 / 엉금엉금 이동
============================================================

다운 상태:
- 폭발 판정으로 선택된 플레이어는 쓰러진다.
- 다운 플레이어는 바닥에 남아 엉금엉금 이동할 수 있다.
- 다운 이동 속도는 일반 이동보다 훨씬 느리다.
- 다운 상태의 카메라는 본인 캐릭터를 추적한다.

다운 플레이어 제한:
- 대시 불가.
- 도발 불가.
- 아이템 사용 불가.
- 기믹 작동 불가.
- 타겟 후보 제외.
- 매치 승리 후보 제외.

생존자가 다운 플레이어를 밟는 경우:
- 생존자가 다운 플레이어의 밟힘 영역에 들어오면 짧게 넘어지거나 휘청인다.
- 잠시 이동이 제한되고 속도가 크게 줄어든다.
- 다시 일어나기 전까지 조작이 제한될 수 있다.
- 같은 다운 플레이어로 인한 연속 발동을 막기 위해 재발동 쿨타임을 둔다.

폭탄이 다운 플레이어를 밟는 경우:
- 폭탄이 다운 플레이어를 밟으면 잠시 순간 가속한다.
- 영구적으로 빨라지는 것이 아니다.
- 여러 번 연속으로 밟아도 가속이 무한 누적되면 안 된다.

디자인 의도:
- 탈락자가 게임에서 완전히 사라지지 않게 한다.
- 죽은 플레이어도 웃긴 장애물로 남아 장면에 참여한다.
- 단, 탈락자가 생존자의 승패를 과도하게 좌우하면 안 된다.


============================================================
9. 근접 마이크 보이스
============================================================

핵심 아이디어:
- 여기서 보이스는 미리 녹음된 효과음이 아니다.
- 실제 플레이어 마이크 입력을 사용하는 게임 보이스다.
- 항상 음성이 들리는 상시 음성 채팅이 아니라, 폭탄이 플레이어와 가까워졌을 때 보이스가 들리거나 강조되는 구조를 목표로 한다.
- 폭탄이 가까워졌을 때 플레이어가 실제로 “꺄아아악”, “오지마!” 같은 반응을 내면 그 소리가 게임 안에서 재미 요소가 된다.

발동 방향:
- 폭탄과 플레이어 사이 거리가 일정 이하일 때 해당 플레이어의 마이크 보이스가 활성화/강조된다.
- 현재 타겟은 더 높은 우선순위 또는 더 넓은 활성 거리로 처리할 수 있다.
- 비타겟도 폭탄이 가까이 지나가면 잠깐 보이스가 들릴 수 있다.
- 다운 플레이어는 우선순위를 낮게 둘 수 있다.

필수 옵션:
- 마이크 사용 On/Off.
- 내 마이크 음소거.
- 특정 플레이어 음소거.
- 전체 보이스 끄기.
- 보이스 볼륨 조절.

주의:
- 실제 마이크 입력은 악용 가능성이 있으므로 음소거 기능이 반드시 필요하다.
- 보이스가 폭탄 경고음, 타겟 경고음 같은 핵심 게임 사운드를 가리면 안 된다.
- MVP에서는 실제 보이스 솔루션을 나중에 붙이고, 우선 거리 기반 활성 규칙만 설계해도 된다.


============================================================
10. 빠른 대전 대기장 / 프리매치 놀이터
============================================================

역할:
- 빠른 대전에서 매치 인원이 모일 때까지 플레이어가 기다리는 공간이다.
- 단순 로딩/대기 화면이 아니라, 뛰어다니고 장난칠 수 있는 작은 놀이터다.
- 매치가 시작되면 모든 플레이어는 실제 매치 맵으로 이동하거나, 대기장에서 선택된 매치 맵으로 전환된다.

입장 수 표시:
- 현재 입장한 플레이어 수와 목표 인원을 명확히 보여준다.
- 예: “입장 중 5 / 8”
- 새 플레이어가 들어오면 UI 숫자가 갱신된다.
- 입장 알림은 귀엽고 짧게 보여준다.
- 목표 인원이 모이면 카운트다운 후 매치가 시작된다.

대기장 조작:
- 플레이어는 대기장 안에서 이동할 수 있다.
- 대시를 사용할 수 있다.
- 서로 부딪히거나 가볍게 밀리는 충돌 판정은 유지한다.
- 대기장 안에서의 행동은 실제 매치 기록이나 승패에 영향을 주지 않는다.

대기장 장난 요소:
- 대기장에는 여러 투척 아이템이 떨어져 있을 수 있다.
- 예: 뚫어뻥, 바나나 껍질, 고무공, 말랑한 똥 장판, 슬라임 덩어리.
- 플레이어는 아이템을 줍고 던지며 기다릴 수 있다.
- 대기장 아이템 효과는 실제 매치보다 더 가볍고 코믹하게 둔다.
- 대기장에서는 탈락/다운/폭탄 타겟 같은 핵심 승패 판정은 발생하지 않는다.
- 필요하면 작은 연습용 폭탄/더미 오브젝트를 둘 수 있지만, 실제 탈락은 발생하지 않는다.

커스텀 방과의 차이:
- 커스텀 방은 방장이 최대 인원과 시작 조건을 직접 정한다.
- 따라서 커스텀 방은 로비 UI 중심으로 시작해도 된다.
- 커스텀 방에도 놀이터형 로비를 넣을 수 있지만 필수는 아니다.
- 빠른 대전은 인원이 자동으로 모이는 동안 지루함을 줄이기 위해 대기장이 우선 적용된다.

디자인 의도:
- 빠른 대전 대기 시간을 지루하지 않게 만든다.
- 플레이어가 조작감, 대시, 아이템 던지기를 자연스럽게 연습하게 한다.
- 새로 들어온 플레이어 수가 보이면 “곧 시작한다”는 기대감이 생긴다.
- 본 매치 시작 전부터 장난스럽고 시끄러운 파티게임 분위기를 만든다.

구현 메모:
- QuickMatchWaitingScene 또는 PreMatchPlaygroundScene으로 분리할 수 있다.
- 대기장 아이템은 본 매치 아이템 시스템을 재사용하되, 승패 영향이 없도록 별도 규칙을 둔다.
- 입장 인원 UI는 MatchmakingService 또는 RoomStateService의 현재 플레이어 수를 구독한다.
- 목표 인원 도달 후 짧은 카운트다운을 시작한다.
- 카운트다운 중 인원이 빠지면 시작을 취소하거나 유예할 수 있다.
- 네트워크에서는 대기장에서도 플레이어 이동/아이템 장난은 동기화하되, 판정 강도는 낮게 유지한다.


============================================================
11. 맵 / 기믹 / 투척 아이템
============================================================

맵:
- 탑다운 3D 맵.
- 논리적 grid/cell 기반으로 맵 데이터를 구성한다.
- Unity 2D Tilemap은 사용하지 않는다.
- 바닥은 하나의 큰 Plane/Mesh 또는 적은 수의 프리팹으로 구성한다.
- 벽, 장애물, 기믹은 3D 프리팹으로 배치한다.

기본 맵 기믹:
- 일방통행 파이프.
- 스위치/셔터.
- 컨베이어 벨트.
- 범퍼 패드.
- 젤리/미끄러운 바닥.
- 회전문.
- 랜덤 박스.
- 폭탄비/메테오 경고.

맵 컨셉:
- 맵은 단순 벽 배치 차이가 아니라 고유한 테마와 대표 기믹을 가져야 한다.
- 예:
  - 바나나 공장: 컨베이어, 바나나 껍질, 미끄러운 바닥.
  - 화장실 대소동: 뚫어뻥, 말랑한 똥 장판, 변기 파이프.
  - 젤리 연구소: 젤리 바닥, 범퍼, 슬라임.
  - 장난감 방: 블록 벽, 고무공, 회전문.
  - 화산 간식섬: 메테오, 뜨거운 바닥, 경고 원.

투척 아이템:
- 플레이어는 한 번에 아이템 1개만 들 수 있다.
- 아이템은 코믹한 방해용이며 직접 킬 수단이 아니다.
- 타겟에게 긴 스턴을 주면 안 된다.
- 아이템 후보:
  - 바나나 껍질.
  - 뚫어뻥.
  - 말랑한 똥 장판.
  - 고무공.
  - 슬라임 덩어리.
- Host/Master가 줍기, 던지기, 충돌, 효과 적용을 확정한다.


============================================================
12. 프로토타입 에셋 원칙
============================================================

핵심 원칙:
- 최종 아트 에셋을 기다리지 않는다.
- 모든 초기 기믹, 프랍, 투척 아이템, 벽, 장애물, 맵 오브젝트는 Unity 기본 Primitive 기반 프리팹으로 먼저 만든다.
- Cube, Sphere, Capsule, Cylinder, Plane과 단순 색상 머티리얼을 사용한다.
- 최종 Blender 제작, 구매 에셋, 폴리싱은 핵심 루프가 검증된 뒤 결정한다.

프리팹 구조:
PrefabRoot
- Logic Components
- Collider
- Visual
  - Cube / Sphere / Cylinder / temporary mesh

목표:
- 나중에 Visual만 교체해도 로직과 Collider가 유지되도록 한다.
- 임시 오브젝트라도 Collider, Layer, 권한 판정은 대충 만들지 않는다.


============================================================
13. 밸런스 데이터화 원칙
============================================================

핵심 원칙:
- 태그 범위, 도발 범위, 폭발 범위, 폭탄 단계 시간, 폭탄 속도 같은 수치는 코드에 고정하지 않는다.
- ScriptableObject 또는 그에 준하는 데이터 에셋에서 조정 가능해야 한다.
- Codex나 개발자는 매직 넘버를 코드 곳곳에 흩뿌리면 안 된다.

데이터화 대상:
- 태그/타겟 전달 범위.
- 태그 면역 시간.
- 플레이어 충돌 반경.
- 플레이어 분리/밀림 강도.
- 다운 플레이어 충돌/밟힘 반경.
- 도발 범위.
- 도발 쿨타임/후딜/충전 시간.
- 폭발 최대 범위.
- 폭발 링 확장 속도.
- 폭탄 평온기 지속 시간 min/max.
- 폭탄 경고기 지속 시간 min/max.
- 폭탄 초폭주기 지속 시간 min/max.
- 폭탄 단계별 기본 속도.
- 폭탄 분노 속도 보정.
- 다운 플레이어 이동 속도.
- 다운 플레이어 밟힘 판정 반경.
- 생존자 밟힘 감속/넘어짐 시간.
- 폭탄 다운 플레이어 밟기 가속 배율/시간.
- 근접 보이스 활성 거리/감쇠/우선순위.

추천 데이터 에셋:
- GameBalanceSettings.
- BombBalanceSettings.
- PlayerBalanceSettings.
- TauntBalanceSettings.
- ExplosionBalanceSettings.
- DownedBalanceSettings.
- VoiceProximitySettings.

초기에는 GameBalanceSettings 하나로 시작하고, 규모가 커지면 분리해도 된다.


============================================================
14. 네트워크 권한 원칙
============================================================

기본 원칙:
- Host/Master가 주요 게임 판정을 확정한다.
- 클라이언트는 입력 요청과 시각/청각 연출을 담당한다.
- 폭발 탈락자, 타겟 변경, 도발 효과, 밟힘 효과, 아이템 효과는 Host/Master 기준으로 확정한다.

Host/Master 확정 대상:
- 폭탄 생성 위치.
- 폭탄 단계별 랜덤 지속 시간.
- 폭탄 타겟.
- 태그 성공/실패.
- 태그 면역 상태.
- 플레이어 위치 보정과 충돌/분리 판정.
- 도발 영역 효과.
- 폭발 후보와 탈락자.
- 다운 플레이어 밟힘 효과.
- 폭탄 임시 가속.
- 아이템 충돌/효과.

클라이언트 역할:
- 입력 전달.
- 로컬 예측 가능한 연출.
- 권위 결과 수신 후 보정.
- 카메라, UI, 사운드, 이펙트 재생.


============================================================
15. UI / 연출 원칙
============================================================

타겟 UI:
- 머리 위 큰 화살표.
- 발밑 레이더.
- 폭탄과 타겟을 잇는 시각적 연결.
- 타겟이 누구인지 즉시 알 수 있어야 한다.

태그 면역 UI:
- 머리 위 원형 게이지.
- 방어막처럼 보이면 안 된다.
- “태그 면역 상태입니다!” 안내 문구.

도발 UI:
- 도발 영역 표시.
- 도발 중 본인 이동/대시 불가 상태 표시.
- 도발 영역 안 플레이어의 Dash 비활성 표시.

폭발 UI/연출:
- 2명 이상이 폭발 범위 안에 있으면 링 연출로 최단거리 탈락자를 보여준다.
- 링이 가장 가까운 플레이어에게 닿는 순간, 해당 플레이어를 명확히 표시한다.
- 이후 줌 아웃과 폭발 넉백으로 모두 날아가되, 탈락자는 1명만 남는다.

폭탄 단계 연출:
- 평온기, 경고기, 초폭주기는 색상, 표정, 소리, 속도, 이펙트로 구분한다.
- 숫자 타이머에 의존하지 않는다.


============================================================
16. 사운드 방향
============================================================

폭탄 사운드:
- 평온기: 낮은 웅웅거림, 귀여운 발소리.
- 경고기: 빠른 비프음, 불안한 리듬.
- 초폭주기: 심장박동, 스파크, 긴장감 높은 사운드.

플레이어 사운드:
- 대시, 넘어짐, 태그, 다운, 기어가기 등은 짧고 코믹하게.
- 근접 보이스는 실제 마이크 입력 기반 시스템으로 별도 취급한다.

우선순위:
1. 폭탄 위험 경고.
2. 타겟 경고.
3. 폭발/링 판정 연출.
4. 태그/태그 면역 피드백.
5. 도발/대시/아이템.
6. UI/보조 사운드.


============================================================
17. 개발 우선순위
============================================================

1차 로컬 코어:
- Bootstrap.
- 로컬 플레이어 이동.
- 대시.
- 테스트 맵/벽 충돌.
- 빠른 대전 대기장 프로토타입.
- 대기장 입장 수 UI.
- 대기장 투척 아이템 장난 프로토타입.
- 폭탄 생성.
- 폭탄 타겟 추격.
- 랜덤 단계 시간과 단계별 속도.
- 태그/태그 면역.
- 폭발 범위와 최단거리 1명 탈락.
- 새 폭탄 생성과 새 타겟 설정.
- 플레이어끼리 겹치지 않는 충돌/분리 판정.
- 최후의 1인 매치 종료.

2차 게임성:
- 도발 홀드/도발 영역/대시 봉인.
- 다운 상태/엉금엉금 이동.
- 다운 플레이어 밟힘 효과.
- 폭탄의 다운 플레이어 밟기 임시 가속.
- 폭발 링 연출.
- 폭탄 재생성 카메라 연출.

3차 확장:
- 투척 아이템.
- 맵 기믹.
- 맵 컨셉 샘플.
- 중간 이벤트 또는 다음 매치 이벤트 투표.
- Bomb Rain/메테오 압박.
- 근접 마이크 보이스 설계/연동.

4차 네트워크:
- 룸/로비.
- Host/Master 권위 판정.
- 타겟/폭탄/폭발/다운/도발 동기화.
- Host 이탈 처리.
- UGC 맵 공유.
